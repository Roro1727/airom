airom ships its rule packs **embedded in the binary**, so a scan always works
with no network. But a new AI framework shouldn't require a new airom release to
detect. `airom rules update` fetches a newer, **signed** rule bundle from the
[airom-rules](https://github.com/airomhq/airom-rules) channel into the cache;
scans then prefer it — while the embedded packs remain the offline floor.

```bash title="Update, then scan"
airom rules update           # fetch + verify + cache the latest bundle
airom fs .                   # this scan now uses it
```

## The model

<Steps>
  <Step title="Embedded packs are the floor">
    airom always has a complete rule set compiled in. With no bundle fetched (or
    with `--no-cached-rules`), that's what every scan uses. Nothing here requires
    a network or a second install.
  </Step>
  <Step title="`rules update` fetches a bundle — explicitly">
    Only this command touches the network. It resolves the release, verifies it,
    and caches it under `<cache-dir>/rules/<version>`. A **scan never fetches**,
    so a scan is reproducible: it uses whatever bundle is already cached.
  </Step>
  <Step title="Scans prefer the cached bundle">
    Once cached, the bundle replaces the embedded packs as the base rule layer.
    Your `--rules` overlays still layer on top. Every AIBOM records which ruleset
    produced it (see [Provenance](#provenance)).
  </Step>
</Steps>

## Pinning and reproducibility

`rules update` with no argument installs the latest release; pass a version to
pin. In CI, pin **both** airom and the rules for a byte-reproducible AIBOM:

```bash title="Pin the tool and the rules"
pip install airom==<version>
airom rules update v1.2.0
airom fs . -o cyclonedx=aibom.json
```

Because a scan never fetches, the same cached bundle yields the same output every
time. `rules update` is the one deliberate act that changes your ruleset.

## Offline and air-gapped

<Note>
  A scan is offline by default — it never reaches the network for rules. Only
  `rules update` does, and `--offline` refuses it before dialing.
</Note>

```bash
airom rules update --offline    # error: refuses to fetch
airom fs . --offline            # fine: uses the cached bundle (or embedded packs)
```

For an air-gapped environment, run `rules update` where there is network, then
copy the cache directory (`<cache-dir>/rules`) to the isolated host, or point
`--rules-source` at an internal mirror that serves the release assets.

## Trust

A bundle is a gzipped tar of the packs plus a `manifest.json` (version, tarball
SHA-256, counts) and a detached **ed25519 signature** over the manifest. airom
verifies, in order:

1. the signature against the **public key embedded in the binary**, then
2. the tarball's SHA-256 against the signed manifest, then
3. extracts (path-traversal-guarded, size-bounded).

Any failure is **fatal** and leaves the previously active bundle untouched. The
signing key's private half lives only in the airom-rules CI secret; rotating it
means a new airom release with the new public key.

<Warning>
  `--insecure-skip-signature` bypasses signature verification (the checksum is
  still enforced). Use it only against a source you fully control — never the
  public channel.
</Warning>

## Falling back and cleaning up

```bash
airom fs . --no-cached-rules    # ignore the bundle; use the embedded packs
airom clean                     # remove the cache, including fetched bundles
```

A cached bundle that fails to load never bricks a scan — airom falls back to the
embedded packs with a warning.

## Provenance

Every AIBOM records the active ruleset, so a document is traceable to the rules
that produced it:

```json
"tool": {
  "name": "airom",
  "version": "v1.3.0",
  "rulesVersion": "v1.2.0",
  "rulesHash": "5f9eef7d107acbdf…"
}
```

`rulesVersion` is `builtin` or the fetched bundle version; `rulesHash` is the
SHA-256 of the effective ruleset. Both project into CycloneDX
(`airom:rules.version` / `airom:rules.hash`) and SARIF.

## Authoring rules

Rules live in [airom-rules](https://github.com/airomhq/airom-rules); stable ones
are promoted into airom's embedded packs over time. To write one, see
[Writing rules](/rules/writing-rules) — the schema and fixtures are identical to
the built-in packs.

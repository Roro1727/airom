Beyond inventory, AIROM surfaces an **artifact-risk overlay**: structural,
statically-detected properties of a model artifact — or the code that loads one — that
enable code execution or content injection at load time. A poisoned checkpoint, an unsafe
deserialization surface. Risks are not a separate security scan bolted on the side: every
risk **attaches to a component already in the AIBOM** and carries the `file` and offset
that prove it.

<Warning>
**A risk is suspicion with evidence, never a verdict.** Static analysis is evadable by
construction, so the *absence* of risks is not a safety claim — and a flagged risk is not
a malware conviction. Read it as "load this in a sandbox and look," not "this is malware."
</Warning>

## The catalog

Severity is a **fixed function of the risk id** — never a judgment made at scan time — so
output is deterministic.

| Risk (`--fail-on` slug) | Severity | What it catches |
|---|---|---|
| `pickle-import` | high | A pickle `GLOBAL` in a Torch checkpoint resolving a code-execution callable (`os.system`, `subprocess`, `builtins.eval`, …). Python runs these *while unpickling*, so the code fires before any model exists. |
| `keras-lambda` | high | A Keras `.h5`/`.hdf5` config declaring a `Lambda` layer — marshalled Python that `load_model` executes while reconstructing the graph. |
| `gguf-template` | medium | A GGUF `tokenizer.chat_template` carrying Jinja sandbox-escape gadgets (`__globals__`, `__subclasses__`, `os.popen`, …) — the reach a template-injection payload uses. |
| `savedmodel-pyfunc` | medium | A TensorFlow SavedModel graph invoking a `PyFunc`-family op (`PyFunc`, `PyFuncStateless`, `EagerPyFunc`) — a registered Python callback in the graph. |
| `unsafe-load` | medium | A `torch.load(..., weights_only=False)` call site — an explicit opt-out of PyTorch's safe path. A **bare** `torch.load(...)` is deliberately not flagged (torch ≥ 2.6 is safe-by-default). |

Every risk carries a `detail` list naming the exact evidence — the dotted callables, the
matched gadgets, the specific op — plus the occurrence that located it.

<Info>
Four of the five inspect an **artifact** (the bytes of a model file). `unsafe-load` is a
**code** risk: it rides on the library the call uses (`torch`), with the call's
`file:line` as evidence. That path is extensible — see [Extending the
catalog](#extending-the-catalog).
</Info>

## How risks appear in output

<Tabs>
  <Tab title="CycloneDX">
    Risks project into the top-level `vulnerabilities[]` array — a non-CVE `id` with
    `source.name: airom`, `ratings[].method: other` (**no CVSS is fabricated**), and
    `affects[].ref` pointing at the affected component's `bom-ref`.

    ```bash title="CycloneDX with a vulnerabilities[] block"
    airom fs . -o cyclonedx=airom.cdx.json
    ```
  </Tab>
  <Tab title="SARIF">
    Each risk becomes a `risk/<slug>` rule carrying GitHub's `security-severity` marker,
    and a result (`error` / `warning` by severity) on the affected artifact — so a
    poisoned checkpoint shows up as a **security alert on the PR that introduced it**.

    ```bash title="SARIF for code scanning"
    airom fs . -o sarif=airom.sarif
    ```
  </Tab>
  <Tab title="Native (json / yaml)">
    Lossless: `component.risks[]` as `{ id, severity, detail, occurrence }`.

    ```bash title="Full-fidelity risks"
    airom fs . -o json=airom.json
    ```
  </Tab>
  <Tab title="Table / CI gate">
    The table view adds a `RISK` column. The CI gate matches `risk` (any),
    `risk:<severity>`, or `risk:<slug>`:

    ```bash title="Fail the build on a high-severity risk"
    airom fs . --exit-code 1 --fail-on "risk:high"
    airom fs . --exit-code 1 --fail-on "risk:unsafe-load"
    ```
  </Tab>
</Tabs>

<Note>
Legacy `airom:pickle.risk` / `airom:pickle.imports` component properties are still emitted
alongside the CycloneDX `vulnerabilities[]` entry for one release, then retire.
</Note>

## Coverage bounds

<Warning>
Artifact-risk scans read a **bounded** prefix of each file — the same `--max-file-size`
cap (default 1 MiB) every content detector uses, so peak memory stays a function of
configuration, not input size. A risk signal placed *beyond* that bound is not inspected:
a GGUF `chat_template` pushed past the cap by a large preceding vocabulary, or a `Lambda`
buried deep in an oversized Keras config, can evade the scan while a runtime that reads the
whole file still executes it. Raise `--max-file-size` to widen the window. A clean scan of
a capped read means "nothing dangerous in the inspected prefix," not "safe."
</Warning>

## Extending the catalog

Code-level risks extend without Go. Any rule pack can attach a catalog risk to a match with
a `risk:` field:

```yaml title="A rule that attaches a risk"
- id: unsafe-load/torch-weights-only-false
  kind: library
  provider: pytorch
  keywords: ["torch.load"]
  pattern: '...'                    # matches torch.load(..., weights_only=False)
  regions: [code]
  claim: { name: "torch" }
  risk: AIROM-RISK-UNSAFE-LOAD      # ← attaches the catalog risk to the match
```

The risk id is validated against the catalog at load time, so a typo fails `airom rules
lint` like any other error. See [writing rules](/rules/writing-rules).

<Info>
It stays **deterministic and offline** — no LLM, no vulnerability database, no network. The
same bytes produce the same risks on every run.
</Info>
